1. Project Overview

Swiggybot is a conversational AI chatbot that lets users order real food from Zomato through natural language. Instead of navigating the Zomato app, users simply chat — "show me biryani restaurants", "add 2 Poori", "checkout" — and the bot handles everything end to end: searching restaurants, displaying menus, building a cart, creating an order on Zomato, and tracking delivery.

Key Highlights

  • Real Zomato Integration — Uses Zomato's official MCP (Model Context Protocol) server for live restaurant search, menus, cart creation, checkout, and order tracking.
  • AI-Powered Conversation — Google Gemini 2.0 Flash understands natural language, extracts intent (cuisine, budget, veg preference), and drives the conversation.
  • Two Client Channels — Web chat (Next.js frontend) and Telegram bot.
  • Stateful Sessions — File-persisted sessions survive server restarts; each user's cart, location, and order state are maintained.
  • Full Order Lifecycle — Search → Menu → Cart → Address → Payment → Confirmation → Place Order → Track → Cancel.

5. Backend — Step-by-Step Working

Step 1: Server Startup (main.py)

When you run python3 main.py:

  1. Environment variables loaded — load_dotenv() reads .env file (contains GEMINI_API_KEY, TELEGRAM_BOT_TOKEN).
  2. FastAPI app created with CORS middleware allowing all origins (for the Next.js frontend).
  3. Lifespan handler starts the Telegram bot as a background task via asyncio.create_task(telegram_bot.start()).
  4. Routers mounted — /api/chat and /api/restaurants.
  5. Server runs on port 8000 using Uvicorn with hot-reload.
# main.py — simplified
@asynccontextmanager
async def lifespan(app: FastAPI):
    asyncio.create_task(telegram_bot.start())  # Telegram runs in background
    yield
    await telegram_bot.stop()

app = FastAPI(title="AI Restaurant Order Bot", lifespan=lifespan)
app.add_middleware(CORSMiddleware, allow_origins=["*"], ...)
app.include_router(chat_router)      # /api/chat/*
app.include_router(restaurant_router) # /api/restaurants/*

Step 2: Receiving a Chat Message (routes/chat.py)

When the frontend sends a POST to /api/chat/message:

  1. The request body is validated by Pydantic: ChatMessage(message, user_id, user_name, user_location, filters)
  2. The handler calls gemini_agent.process_message() with a 28-second timeout
  3. Response includes: bot text, conversation state, cart items, restaurant info, order data, and thinking steps
  4. Sessions are saved to disk after every message via session_service.save()
# routes/chat.py
@router.post("/message", response_model=ChatResponse)
async def send_message(chat: ChatMessage):
    response, thinking_steps = await asyncio.wait_for(
        gemini_agent.process_message(
            user_id=chat.user_id,
            message=chat.message,
            user_location=chat.user_location,
            filters=chat.filters,
        ),
        timeout=28,  # Render has a 30s limit
    )
    session = session_service.get_session(chat.user_id)
    return ChatResponse(response=response, state=session.state.value, ...)

Step 3: The Gemini Agent Processes the Message (services/gemini_agent.py)

This is the brain of the entire application — a 1900+ line Python class. Here's what happens inside process_message():

3a. Session & Location Setup

  1. Gets or creates user session from SessionService
  2. Updates location if user provided one (via GPS or text)
  3. Clears cached address_id if location changed (so Zomato re-resolves the correct delivery address)
  4. Adds the user's message to conversation history (kept at last 20 messages)

3b. Intent Detection (No LLM — Pure Python)

Before calling Gemini, the agent tries to handle the message with fast pattern matching. This avoids slow LLM calls for common actions:

CheckMethodWhat It Does
State-based interceptsState machine checksCONFIRMING_ORDER → intercept yes/no; AWAITING_ADDRESS → capture address; AWAITING_PAYMENT → capture payment choice
Location update_is_location_update_only()Messages like "Amaravati, Andhra Pradesh" → save location, skip LLM
Smalltalk_is_smalltalk_request()"hi", "hello", "help" → instant welcome message
Cart viewExact match"show cart", "view cart" → render cart contents
Order tracking_is_order_tracking_request()"track my order", "order status" → call Zomato get_order_tracking_info
Cancel order_is_cancel_order_request()"cancel order" → inform user to cancel via Zomato app (no MCP cancel tool exists)
Session resetExact match"start over", "reset" → clear session
Restaurant selection_is_restaurant_selection_request()"1", "2" or restaurant name → load that restaurant's menu
Restaurant search_is_restaurant_search_request()"show me biryani restaurants" → call Zomato search directly
Key Design Decision: About 80% of messages are handled by fast Python pattern matching without calling the Gemini LLM. This makes the bot much faster (instant vs. 2-3 seconds) and cheaper (no API calls for simple actions).

3c. Restaurant Search (Direct MCP Call)

When a search is detected (e.g., "show me biryani restaurants"):

  1. Extract keyword — _extract_search_keyword("show me biryani restaurants") → "biryani"
  2. Resolve Zomato address_id — Calls get_saved_addresses_for_user MCP tool, then fuzzy-matches user's location against saved Zomato addresses
  3. Call MCP — get_restaurants_for_keyword(keyword="biryani", address_id="...", page_size=10)
  4. Extract restaurant data — Walks the nested JSON to find restaurant IDs, names, ratings, delivery times, cuisines
  5. Rank by relevance — Sorts results so restaurants whose name or cuisine matches the keyword appear first (e.g., "Hyderabad Biryani House" before "Hello Idly")
  6. Format & return — Numbered list with name, rating, delivery time, cuisine

3d. Menu Loading

When user selects a restaurant (e.g., replies "1"):

  1. Two-step MCP menu fetch:
    • get_menu_items_listing(res_id, address_id) → gets categories + variant_id mappings
    • get_restaurant_menu_by_categories(res_id, categories, address_id) → gets items with prices
  2. Extracts: item name, price, size, variant_id (critical for ordering), veg/non-veg tags
  3. Stores items in session.menu_items_map for cart operations
  4. Formats as a clean menu list and returns

3e. Order Item Parsing & Cart Building

When user says "2 Poori and 1 Masala Dosa":

  1. _parse_order_items() uses regex to extract: [("poori", 2), ("masala dosa", 1)]
  2. Fuzzy-matches each item name against the loaded menu items
  3. Creates CartItem objects with the correct variant_id from the menu
  4. Adds to session cart, transitions to ORDERING state
  5. Returns a cart summary with the "checkout" prompt

3f. Checkout & Order Placement

When user says "checkout" → "yes":

  1. Collects delivery address (AWAITING_ADDRESS state)
  2. Collects payment method — UPI QR or Pay Later/COD (AWAITING_PAYMENT state)
  3. Shows order summary and asks for confirmation (CONFIRMING_ORDER state)
  4. On "yes": calls Zomato MCP create_cart(res_id, items, address_id, payment_type)
  5. Then calls checkout_cart(cart_id) to finalize the order
  6. Parses the response for order_id, ETA, total
  7. Returns success message with order details

3g. Fallback to Gemini LLM

If none of the above pattern-matching handlers can process the message, it falls through to the Gemini LLM:

  1. Builds a system prompt with location context, active filters, and conversation history
  2. Converts Zomato MCP tools into OpenAI function-calling format using _sanitize_schema()
  3. Sends to Gemini via OpenAI-compatible API with the tools
  4. If Gemini returns tool calls → executes them against MCP → feeds results back
  5. Up to 4 tool-calling loops allowed (to handle multi-step queries)
  6. Final text response is returned to the user

7. Zomato MCP Integration

What is MCP?

Model Context Protocol (MCP) is an open protocol that lets AI models call external tools in a standardized way. Zomato provides an MCP server at https://mcp-server.zomato.com/mcp that exposes real Zomato functionality as callable tools.

How the Connection Works (zomato_mcp.py)

  1. Uses npx mcp-remote@0.1.37 to establish a local stdio bridge to the remote MCP server
  2. Connects via mcp.client.stdio.stdio_client → creates a ClientSession
  3. Discovers available tools via session.list_tools()
  4. Has timeout protection: 12s connect timeout, 10s per tool call, 60s cooldown on failure
  5. Singleton instance global_zomato_mcp shared across all requests

Schema Sanitization

Zomato MCP tool schemas are complex and may contain anyOf, oneOf, and additionalProperties that the Gemini/OpenAI API doesn't understand. The _sanitize_schema() method cleans these schemas:

  • Resolves anyOf/oneOf → picks first non-null variant
  • Removes additionalProperties, default, $schema
  • Truncates descriptions to 120 chars (saves tokens)
  • Removes address_id from required params (agent resolves it separately)

8. Gemini AI Agent — Core Logic

Agent Initialization

# Uses OpenAI SDK with Gemini's compatible endpoint
client = AsyncOpenAI(
    api_key=GEMINI_API_KEY,
    base_url="https://generativelanguage.googleapis.com/v1beta/openai/"
)
# Models with automatic fallback
models = ["gemini-2.0-flash", "gemini-1.5-flash"]

System Prompt Design

The system prompt tells Gemini it's a food-ordering assistant and provides:

  • How to use each MCP tool (with examples)
  • Current user location context
  • Active search filters (veg, rating, distance)
  • Rules: never make up data, always confirm before ordering, keep responses concise

Tool Calling Flow

1. Build messages array: [system_prompt, ...conversation_history, user_message]
→ 2. Call Gemini with tools=openai_tools (converted from MCP schemas)
→ 3. If Gemini returns tool_calls:
    a. Execute each tool via global_zomato_mcp.call_tool(name, args)
    b. Add tool results to messages
    c. Call Gemini again with the augmented messages
→ 4. Repeat up to 4 loops (MAX_TOOL_LOOPS)
→ 5. Return final text response

Model Fallback & Error Handling

  • Multi-model fallback: If gemini-2.0-flash fails (rate limit, 500 error), automatically retries with gemini-1.5-flash
  • 14-second timeout per LLM call to stay within Render's 30s limit
  • Tool result truncation: MCP results are capped at 700 chars to avoid token bloat
  • History trimming: Only last 5 messages with max 500 chars each are sent to the LLM

14. Interview Q&A — How to Explain

Q: "Tell me about this project."

A: "I built an AI food ordering chatbot called Swiggybot. It integrates with Zomato's real MCP server to let users search restaurants, browse menus, build a cart, and place actual orders — all through natural language conversation. The backend is FastAPI with Google Gemini AI for understanding user messages, and I have a Next.js web chat frontend plus Telegram bot support."

Q: "How does the backend work?"

A: "The FastAPI backend has a single entry point at /api/chat/message. When a message arrives, the Gemini Agent first checks if it can handle the request through fast pattern matching — things like greetings, cart views, order tracking, or restaurant selection. About 80% of messages are handled this way, making the response instant. For complex requests, it falls through to the Gemini LLM, which can use Zomato MCP tools — search restaurants, load menus, create carts, and checkout. The conversation runs on a finite state machine with 10 states from IDLE to TRACKING."

Q: "What is MCP and why did you use it?"

A: "MCP stands for Model Context Protocol — it's an open standard for AI models to call external tools. Zomato provides an MCP server that exposes 11 tools including restaurant search, menu loading, cart creation, checkout, and tracking. I use it because Zomato doesn't have a public REST API. The MCP client connects via stdio transport using a Node.js proxy (mcp-remote), discovers available tools at startup, and calls them with JSON arguments."

Q: "How do you handle state management?"

A: "Each user has a UserSession Pydantic model that tracks their conversation state, location, cart items, selected restaurant, menu data, and order info. Sessions are stored in-memory for fast access, and persisted to a JSON file after every message so they survive server restarts. The session has a 10-state finite state machine that determines which input handlers are active — for example, in CONFIRMING_ORDER state, only 'yes' and 'no' are accepted."

Q: "How do you ensure fast responses?"

A: "Three strategies: First, I use pattern matching before the LLM — regex and string matching handle greetings, cart management, location updates, and restaurant selection in under 10ms. Second, I cache the Zomato address_id on the session so I don't re-fetch addresses on every request. Third, I have strict timeouts — 14 seconds per LLM call, 10 seconds per MCP tool call, and 28 seconds total per request — with multi-model fallback if the primary model is slow."

Q: "What was the hardest technical challenge?"

A: "The hardest part was making the Zomato MCP integration work end-to-end. The MCP tool schemas had anyOf and nullable types that broke OpenAI's function calling format, so I built a schema sanitizer. The restaurant search returned all nearby restaurants regardless of keyword, so I added keyword relevance ranking. Order tracking returned deeply nested JSON that I had to walk to find the right data. And there's no cancel order tool — I had to detect the intent and guide users to the Zomato app."

Q: "Why FastAPI instead of Django or Flask?"

A: "FastAPI is async-first, which is critical because every request involves multiple async operations — MCP tool calls, LLM API calls, and session persistence. It also gives me automatic request validation via Pydantic, auto-generated Swagger docs at /docs, and native CORS middleware. Django would be overkill and Flask doesn't have native async support."

Q: "How would you scale this?"

A: "Currently sessions are in-memory with file backup, which works for a single instance. To scale: (1) Move sessions to Redis for shared state across instances, (2) Use a message queue like Celery for long-running MCP calls, (3) Add proper database (PostgreSQL) for order history and analytics, (4) Rate limit per user to prevent abuse, (5) Connection pooling for MCP with multiple client instances."

Q: "How do you handle errors?"

A: "Multiple layers: The MCP client has connect/tool timeouts with cooldown on failure. The LLM has multi-model fallback — if Gemini 2.0 Flash fails, it tries 1.5 Flash. The chat endpoint has a 28-second timeout returning a 504 if exceeded. Every state-handler has try/except that logs errors and returns user-friendly messages like 'I had trouble reaching Zomato, please try again.' The frontend shows error styling and a retry button."

Q: "Explain the order placement flow technically."

A: "When the user confirms 'yes' in CONFIRMING_ORDER state: First, I build a list of cart items with their variant_ids (captured from the menu fetch step). Then I resolve the Zomato address_id by fuzzy-matching the user's location against their saved Zomato addresses. I call create_cart(res_id, items, address_id, payment_type) which returns a cart_id. Then I call checkout_cart(cart_id) to finalize the order. Both responses are parsed for errors and order info. On success, the state moves to ORDER_PLACED and the order ID is saved."
